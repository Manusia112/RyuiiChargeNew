вЂ‹1. Database Context (The Root Cause Area)
We have two main structures for products:
вЂ‹products (Table): Stores the raw item data from the provider. Contains the base/modal price (selling_price). Example: selling_price is 900.
вЂ‹products_public (View/Table): Contains the data exposed to the frontend. It calculates and adds a profit margin to the base price. Example: displayed price is 1000.

вЂ‹User selects a game, enters their Player ID, and selects a payment method.
вЂ‹The frontend displays the final price fetched from products_public (e.g., Rp 1.000).
вЂ‹When the user clicks "Bayar Sekarang" (Pay Now), the frontend sends a POST request to the Supabase Edge Function create-order.
вЂ‹The payload likely includes the product_id (or sku), the expected final price (1000), customer details, and payment method.
вЂ‹3. Backend Logic & The Bottleneck (supabase/functions/create-order/index.ts)
вЂ‹The create-order function receives the request.
вЂ‹[THE BUG IS HERE]: The function performs a security validation to ensure the price hasn't been tampered with. However, it currently queries the base products table instead of products_public.
вЂ‹It compares the frontend price (1000) with the database base price (900). Because 1000 != 900, the validation fails.
вЂ‹As a result, the function throws a 404 Not Found error with the message: "Produk tidak ditemukan atau tidak aktif".
вЂ‹Expected behavior after validation: If validation passes, it should insert a record into the orders/transactions table and call the Midtrans API to generate a payment token/QRIS URL.
